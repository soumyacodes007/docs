# Architecture

PIXA is intentionally split into a few layers:

- the MCP package that agents and Claude Desktop load
- the hosted or local browser flows for approvals
- the PIXA Hub treasury and payment router
- optional external services such as on-ramp providers and DEXs
- future AC2-style authorization for delegated sessions

## 1. System Architecture

<Frame>
  <img src="/images/pixa-architecture-system.svg" alt="PIXA system architecture placeholder" />
</Frame>

PIXA starts with a user-funded Algorand wallet and fans out from there.

- the MCP server exposes wallet and payment tools
- the agent uses those tools to inspect balance, pay APIs, and launch flows
- the hub handles treasury-backed payment routing when a service is not on Algorand
- the browser handles approval-heavy flows like on-ramp and rekey

## 2. Normal Payment Flow

<Frame>
  <img src="/images/pixa-architecture-payment.svg" alt="PIXA normal payment flow placeholder" />
</Frame>

The normal flow is simple:

1. the agent requests a tool
2. PIXA checks funds and policy
3. the wallet signs the payment authorization
4. the service verifies the payment and returns the resource

This is the path for x402-style API access and other straightforward wallet operations.

## 3. Multichain Payment Flow

<Frame>
  <img src="/images/pixa-architecture-multichain.svg" alt="PIXA multichain payment flow placeholder" />
</Frame>

PIXA is built around an Algorand home wallet, but the point is not to stay locked to one chain.

For non-Algorand payments:

- the user funds the Algorand side
- PIXA Hub receives or records the payment intent
- the hub routes value to the target chain or service path
- the user still sees one wallet and one control surface

This is why the docs should describe PIXA as a payment control plane, not just a single-chain wallet.

## 4. Multi-Facilitator Topology

<Frame>
  <img src="/images/pixa-architecture-facilitators.svg" alt="PIXA multi-facilitator topology placeholder" />
</Frame>

PIXA can support more than one execution path behind the scenes.

That matters because:

- different chains have different settlement speeds
- different services have different trust models
- some flows should use a fast operator-backed path
- others should move toward more explicit on-chain policy or partner-based routing

The user should not need to understand every rail in the middle.
They should only need to understand what they approved.

## 5. AC2 Session Model

<Frame>
  <img src="/images/pixa-architecture-ac2.svg" alt="PIXA AC2 session model placeholder" />
</Frame>

AC2 is the future control layer for sensitive actions.

In that model:

- the agent requests authority
- the user approves through a wallet or hosted page
- the session is time-bound or spend-bound
- the authority can be revoked later without giving away the main key

That is the cleanest path to safer agent control.
